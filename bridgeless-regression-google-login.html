Mixpanel event drop — Expo 53 → 55 bridgeless flow

Why identical client code now silently drops analytics events on Android
TL;DR

The same line — InteractionManager.runAfterInteractions(cb) — used to mean "fire on next JS tick" under the legacy bridge. Under RN 0.82+ bridgeless mode (forced in Expo 55), it now means "schedule at idle priority via the React scheduler". Idle work gets cancelled or starved when the host pauses (Credential Manager dialog) or when the JS thread is busy (post-OAuth recovery + navigation). The Mixpanel SDK never receives the event. No error. No Sentry capture. Funnel just drops.

When the host resumes and cb finally runs, the call enters a torn-down module proxy. Either the buffer was discarded on invalidate(), or the proxy silently no-ops. No exception thrown.

⚠️ The same mechanism applies to the "Sign in success" event
That event fires right after the OAuth response returns — the JS thread is at peak load (state updates, navigation pending, home screen mount, WebView init). Idle work scheduled here is even more likely to be starved than the "Login button clicked" event.

The two vulnerable points in the typical login flow

handleGoogleSignIn = () => {
  oAuthRegisterHandler({ flow });   // ← Credential Manager dispatched, Activity begins pausing
  trackLoginButtonPress({ ... });   // ← Mixpanel event queued AFTER, host already pausing
}

Risk: medium. ~5–20% drop on Android, depending on device + timing. 

onSuccess: (data) => {
  trackLoginSuccess({ ... });       // ← Queued at idle priority

  // JS thread now extremely busy with:
  //   - Zustand state propagation
  //   - React 19 concurrent re-renders
  //   - socialAuthMutation post-success work
  //   - 100ms setTimeout to navigate
  //   - Home screen mount + WebView load (immediately after)

  setTimeout(() => navigation.navigate(MAIN_STACK.HOME), 100);
  // Login screen unmounts. Idle slot may NEVER get to run.
}

Risk: high. This is the primary suspect for the funnel drop. The idle slot competes with everything happening in the post-OAuth recovery window.

Upstream evidence

The bug matches the exact failure pattern documented in open RN issues:

  • facebook/react-native#54534 — "RN 0.82 – JS Timers & Fetch Freeze After Launching a Non-React AppCompatActivity". The Credential Manager bottom sheet IS the "non-React AppCompatActivity" trigger.
  • facebook/react-native#50327 — "Bridgeless mode: async JS 'stacking up' when MainActivity is in the background". User explicitly reports disabling New Arch fixes it.
  • #47436 + #47437 — setTimeout + fetch broken in bridgeless interop on background.
  • mixpanel/mixpanel-react-native#209 — "Support new React Native architecture" — still open since Aug 2023. The SDK has no codegen, so it runs through the broken interop layer.

How to test the broken flow locally

  1. Use a non-Pixel device if possible (Samsung One UI / Xiaomi MIUI / OnePlus). Pixel is too lenient.
  2. Build a release variant (./gradlew assembleRelease), not debug.
  3. Enable Developer Options → "Don't keep activities". This single toggle reliably surfaces the bug.
  4. Optional: throttle network with Charles / Proxyman to a 3G profile, widening the flush window.
  5. Tap "Sign in with Google" → complete or dismiss → check whether the Mixpanel event arrives within ~30s.
  6. Repeat 5–10 times. ~70–90% of attempts should fail with "Don't keep activities" on.
  7. Toggle the option off and repeat → events should now arrive reliably.

Why this matters: the prod drop is ~27% on Android, meaning natural production timing hits the bad window 27% of the time. Slow human dev testing barely hits it; the toggle forces 100% reproduction.

Proposed fix on our side

1. Drop the InteractionManager wrapper

In modules/app/src/hooks/utility/useTrackMixPanelEvent.ts:

// before:
InteractionManager.runAfterInteractions(() => {
  trackMixPanelEvent({ eventName, params: mergedProperties });
});

// after:
trackMixPanelEvent({ eventName, params: mergedProperties });

Why this works on both versions:

  • Expo 53: direct call enters the bridge → native buffer → no worse than before.
  • Expo 55: direct call skips the React scheduler entirely → goes through the TurboModule interop while the proxy is still valid (before any invalidate() can run).

2. Add mixpanel.flush() on AppState background

Belt-and-suspenders: explicitly drain the native buffer when the app backgrounds, before any module teardown can happen.

AppState.addEventListener('change', (next) => {
  if (next === 'background') {
    Mixpanel.getInstance(...).flush();
  }
});

Mixpanel docs actually recommend this for React Native; it just wasn't necessary in bridge mode because the OkHttp worker did it for you.
